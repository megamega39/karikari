# Karikari 開発 TODO & 改善提案（詳細精査版）

以下の内容は、コードベースを詳細に精査した結果に基づく技術的な指摘と、今後の Claude による実装に向けた提案です。

---

## 1. 技術的指摘（指摘事項）

### ● UIスレッドのブロック（パフォーマンス）
- **書庫読み込みの同期実行**: `nav.cpp` の `LoadArchiveToList` や、フォルダ移動時の処理が一部UIスレッドで行われており、低速なストレージや大量のファイルがある場合にメインウィンドウがフリーズする要因となっている。
- **同期IO**: `NavigateTo` において、フォルダ属性の取得や書庫判定の一部が同期的に行われている。

### ● メモリ管理とアニメーションの脆弱性
- **GIF/WebPデコードの非効率性**: `viewer.cpp` のアニメーション再生ループにおいて、変換用オブジェクト（`IWICFormatConverter`）やピクセルバッファを毎フレームヒープから確保している。これが「最小化時のメモリ異常消費（99%）」の遠因である可能性が高い。
- **デコーダのキャッシュ**: 現在、GIF/WebPの全フレームデータがWIC内部でどのように保持されているか不透明。巨大なファイルの展開時にメモリリークに近い挙動を示す可能性がある。

### ● 基盤システムの課題
- **簡易JSONパーサーの限界**: `settings.cpp` の自作パーサーは `wstring::find` ベースで非常に脆弱。エスケープ文字の処理やネスト構造、書式エラーへの耐性が低いため、将来的な設定項目の拡張に耐えられない。
- **グローバル状態の肥大化**: `g_app` 構造体に全ての状態（ウィンドウ、ナビ、ビューアー、キャッシュ）が詰め込まれており、モジュール間の依存関係が複雑化している。

### ● UI/UXの不備
- **グリッド表示の硬直**: `LVS_AUTOARRANGE` スタイルの欠如。
- **画像描画パスの劣化**: 以前のバージョンで存在した `ID2D1ImageSourceFromWic` による高品質・短レイテンシな描画パスが現在失われており、標準の `ID2D1Bitmap` にフォールバックしている。

---

## 2. 実装提案（Claude への指示項目）

### ⚡ パフォーマンス・メモリ改善（最優先）
- [ ] **アニメーションタイマーの最適化**: 最小化時（`IsIconic`）に `KillTimer` を実行し、復帰時に再開する。
- [ ] **デコードリソースのメンバー化**: `ViewerState` に作業用 `IWICFormatConverter` 等を保持し、再生中は `Initialize` のみで再利用する。
- [ ] **Image Source API の復元**: `decoder.cpp` で `CreateImageSourceFromWic` を使用し、GPU側での YCbCr デコードを有効化する。

### 🎨 UI・UX の洗練
- [ ] **グリッド表示のレスポンシブ化**: `ListView` に `LVS_AUTOARRANGE` スタイルを適用し、リサイズ時に `LVM_ARRANGE` が正しく動作するように調整。
- [ ] **高品質スケーリングの保証**: `DownscaleSource` 内の補間モードを物理メモリやGPU負荷に応じて最適化。

### 🛠 アーキテクチャ強化（リファクタリング）
- [ ] **JSONパーサーの刷新**: より堅牢な（あるいは標準的なライブラリを使用した）パーサーへの置き換え。
- [ ] **WndProc の委譲**: `window.cpp` の巨大な `MainWndProc` を、メッセージ個別のハンドラクラスや関数に分割。
- [ ] **非同期ナビゲーションの統一**: 全てのファイル・書庫アクセスをスレッドプール経由の非同期処理に統一し、UIフリーズを撲滅する。

---

## 3. 追加の検討事項
- **設定 UI の近代化**: 現在の `CreateWindowEx` による手動ダイアログ構築から、リソーススクリプトの活用や近代的なUIライブラリへの移行（長期課題）。
- **テーマ設定の集中管理**: 各所でハードコードされている `RGB(235, 244, 255)` 等の色情報をテーマ定義として一括管理する。
